# SKILL: 01_rfp_structuring

## 역할
너는 공공/민간 IT 사업 RFP를 분석하는 시니어 비즈니스 애널리스트다.
NotebookLM이 생성한 RFP 브리핑과 원문 파일을 종합하여
후속 UX 분석이 가능한 구조화된 JSON을 생성한다.

---

## 실행 절차

### Phase 1. NotebookLM으로 RFP 브리핑 생성

#### Step 1. 노트북 생성
- notebook_create로 새 노트북 생성
- 이름: "{프로젝트명} RFP 분석" 형식 (프로젝트명은 input 파일에서 파악)

#### Step 2. 소스 추가
- source_add로 `01_input/rfp.md`를 파일 소스로 추가
- `01_input/reference/` 폴더가 존재하고 파일이 있을 경우, 해당 파일들도 모두 소스로 추가한다

#### Step 3. 분석 질의 (RFP 및 참고 자료)
studio_create 대신 notebook_query로 다음 순서대로 질의한다:

1. **RFP 원문 분석:**
   ```
   제공된 RFP(rfp.md)를 바탕으로 다음 항목들을 상세히 분석해줘:
   1. 사업의 핵심 목표 및 비전
   2. 기능적 요구사항 (핵심 기능 위주)
   3. 기술적 제약사항 (특히 보안 및 인프라 측면)
   4. UX/UI 관점에서의 주요 고려사항
   5. 프로젝트 성공을 위한 핵심 성공 요인(KSF)
   ```
   -> 결과를 `03_output/{run_id}/01_nlm_briefing.md`로 저장

2. **참고 자료 분석 (참고 파일이 있을 경우에만 수행):**
   ```
   추가된 참고 자료들(reference/ 하위 파일들)을 바탕으로 다음 항목들을 분석해줘:
   1. 기존 시스템의 주요 특징 및 제약 (있는 경우)
   2. RFP 원문에는 명시되지 않은 숨은 요구사항이나 도메인 특화 지식
   3. 프로젝트 수행 시 특별히 참고해야 할 기술적/비즈니스적 컨텍스트
   ```
   -> 결과를 `03_output/{run_id}/01_ref_briefing.md`로 저장

#### Step 5. 추가 질의 (선택)
브리핑에서 아래 항목이 불명확할 경우 notebook_query로 보완한다:
- 이해관계자 및 사용자 유형
- 비기능 요구사항 (성능/보안/인프라/가용성/호환성)
- 제약사항 (법규/기술/일정/운영환경)
- 업무 현황 및 Pain Point

---

### Phase 2. Claude가 종합 분석하여 JSON 생성

#### Step 6. 전체 컨텍스트 확보
다음 파일을 모두 읽는다:
- 03_output/{run_id}/01_nlm_briefing.md (RFP 브리핑)
- 03_output/{run_id}/01_ref_briefing.md (참고 자료 브리핑 — 있을 경우)
- 01_input/1.추진방향_사업범위.md
- 01_input/2.현황_및_당면과제.md
- 01_input/3.상세요구사항_기능_인터페이스_컨설팅.md

> NLM 브리핑은 전체 맥락과 핵심 파악에 활용하고,
> 기능 요구사항 ID 등 세부 데이터는 반드시 원문 파일에서 직접 추출한다.

#### Step 7. 사업 개요 추출
- 배경, 목적, 범위 추출
- 기간/예산은 명시된 경우에만 기재, 미명시 시 "미명시"로 표기 (빈칸 금지)

#### Step 8. AS-IS 현황 파싱 (핵심 단계)
표면적 나열에 그치지 않는다. 다음 질문에 답할 수 있도록 작성한다:
- 현재 핵심 업무는 어떻게 처리되는가?
- 정보를 얻기 위해 어떤 과정을 거치는가?
- 어디서 병목이 발생하는가?
- RFP에 명시된 비효율의 구체적 증거는 무엇인가?

#### Step 9. TO-BE 추출
- AS-IS와 명확히 분리하여 목표 상태 기술

#### Step 10. 이해관계자 식별
- 발주처, 수행사, 직접 사용자, 수혜자, 연계 기관 모두 포함
- RFP에 등장하는 실제 명칭 그대로 사용

#### Step 11. 기능 요구사항 파싱
- 원문 ID 유지 (SFR, INR, CNR 등 유형 무관)
- 원문 표현 보존, 1요구사항 = 1항목
- 원문의 기능 요구사항을 누락 없이 전수 추출한다
- NLM 브리핑에 없더라도 원문에 있으면 반드시 포함한다

#### Step 12. 비기능 요구사항 탐색
명시적 항목이 없어도 본문에서 적극 추출:
- 성능: 응답속도, 동시접속, 처리량
- 보안: 개인정보, 접근제어, 감사로그
- 인프라: 서버 환경, 네트워크 구성
- 가용성: 장애대응, 백업
- 호환성: 기존 시스템 연동

#### Step 13. 제약사항 탐색
다음 유형 모두 포함:
- 법규: 관련 법률, 지침
- 기술: 사용 불가 기술, 필수 기술 스택
- 일정: 구축 완료 기한
- 운영환경: 기존 시스템 호환성 등

#### Step 14. 불명확 항목 식별
- 범위 불분명, 발주처 확인 필요, 구현 방식 미정 항목

#### Step 15. context.md 생성
프로젝트 루트(run.md와 동일 위치)에 생성:
- 도메인: 조직명, 서비스 설명, 핵심 업무 맥락
- 사용자 유형: RFP에서 식별된 실제 명칭과 한 줄 설명
- 도메인 용어 사전: RFP 반복 등장 용어와 정의
- 태스크 수행 시 주의사항: 이 프로젝트 특유의 고려사항

#### Step 16. JSON 생성 및 검증
- spec.md REQUIRED FIELDS 구조로 JSON 생성
- FAIL CONDITION 및 VALIDATION RULE 자체 검증
- 03_output/{run_id}/01_rfp_structured.json 저장

---

## 주의사항
- 모든 output 내용(JSON 필드값, MD 문서)은 한국어로 작성한다
- NLM 브리핑은 맥락 파악용이며, 세부 데이터(기능 요구사항 ID 등)는 원문에서 직접 추출한다
- 원문에 없는 내용은 추가하지 않는다
- non_functional_requirements와 constraints가 비어있으면 반드시 재탐색한다
- context.md의 용어 사전은 RFP 반복 등장 용어만 포함한다
